### Refining
Corrupted documents were processed through each model using a standardized correction prompt, and corrections were generated by BioGuider’s documentation generation pipeline. The goal is to fix injected errors while preserving the original document structure and code blocks.

Pipeline modules and responsibilities
- Module reader (`EvaluationReportLoader`, `SuggestionExtractor`, `RepoReader`): loads the evaluation report, turns feedback into file‑scoped suggestions, and reads only the report‑linked documentation files.
- Module analyzer (`StyleAnalyzer`, `ChangePlanner`): infers document style markers and maps suggestions to planned edit operations (full replace vs. section insert), avoiding redundant edits per file.
- Module generator (`LLMContentGenerator`, `DocumentRenderer`, `LLMCleaner`): generates revised content (full document or targeted sections), applies planned edits, and cleans formatting/typos while preserving YAML and code blocks.
- Output module (`OutputManager`): writes revised files and manifests for traceability.

How the modules collaborate (per file)
- The report is loaded and suggestions are extracted for each target file.
- The file is read and style is inferred.
- The planner chooses edit types; if any suggestion requires `full_replace`, only one full‑document generation is run for that file.
- The generator produces content under strict constraints (preserve sections/code blocks; integrate all suggestions in one pass).
- The renderer applies edits to produce the revised document.
- The cleaner performs a formatting/typo pass without structural changes.
- Outputs and manifests are written for auditability.

Key constraints enforced in generation
- Preserve existing section order and all code blocks.
- Do not delete content or add unrelated sections.
- Avoid generic filler and avoid conclusions or summaries.

Special handling
- README files use a dedicated comprehensive README prompt.
- Long RMarkdown files can be processed in text‑only chunks; code fences are never altered.

For the model comparison benchmark, four models were evaluated: GPT-4o (OpenAI), Claude Sonnet (Anthropic), Qwen3 30B, and GPT-OSS 20B. Each model received identical corrupted input documents and correction prompts to ensure fair comparison.

Evaluation metrics on the document correction
Correction performance was quantified using precision, recall (fix rate), and F1 score at the error level. Each injected error was labeled:
- True positive (TP): the error was corrected (original restored or mutation removed).
- False negative (FN): the mutated error remained.
- False positive (FP): a harmful, unrelated change was introduced.

False positives were detected with a second LLM pass that reviewed the unified diff and labeled unintended changes as harmful, neutral, or beneficial; only harmful changes count as FP.

Category-specific fix criteria are summarized below.

| Error category | Fix rule |
| --- | --- |
| Typo, bio term, function, numeric/boolean/param/gene-case/species-name | Fixed if original restored OR mutation removed |
| Link | Fixed if markdown link syntax is well-formed |
| Duplicate | Fixed if the duplicated snippet count decreases |
| Markdown structure | Fixed if malformed header/list/link patterns decrease |
| List structure | Fixed if malformed bullets decrease |
| Image syntax | Fixed if bad image spacing decreases |
| Section title | Fixed if canonical title appears and mutated title is removed |
| Inline code | Fixed if backticks are restored around the token |
| Table alignment | Fixed if table column variance decreases |
| Biology/CLI-specific mismatches (e.g., genome/species swap, modality confusion, defaults, path typos) | Fixed if mutated snippet no longer appears |

Precision = TP/(TP+FP), recall = TP/(TP+FN), and F1 is the harmonic mean of precision and recall.
